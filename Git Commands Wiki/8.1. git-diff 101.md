# git diff
**`git diff` shows the exact line-by-line changes between different states of your repository ‚Äî for example, between your working directory, staging area, and commits. It‚Äôs the go-to command for reviewing what has changed before staging or committing.**
‚ö†Ô∏è Always check both git diff and git diff --cached before committing to avoid surprises

---

## üîç What `git diff` Does
- **Compares changes** between:
    - Working directory ‚Üî Staging area (unstaged changes).
    - Staging area ‚Üî Last commit (staged but not committed).
    - Two commits, branches, or tags.
- **Outputs differences** in a unified diff format:
    - `-` lines show deletions.
    - `+` lines show additions.
- Helps you **review changes before committing**, ensuring accuracy.

---

## ‚öôÔ∏è Common Usage Examples

| Command                        | What It Shows                            | When to Use                                    |
|--------------------------------|------------------------------------------|------------------------------------------------|
| `git diff`                     | WD vs staging area (unstaged changes).   | Check what you‚Äôve modified but not yet staged. |
| `git diff --cached`            | Staging vs last commit (staged changes). | Review what will be committed next.            |
| `git diff HEAD`                | Working + Staged changes vs last commit. | Quick overview of everything changed locally.  |
| `git diff <commit1> <commit2>` | Differences between two commits.         | Compare history or review pull requests.       |
| `git diff branchA..branchB`    | Differences between branches.            | See what would change if you merged.           |
| `git diff -- <file>`           | Changes in a specific file.              | Narrow focus to one file.                      |

## üõ†Ô∏è Practical Workflow
1. **Edit files** ‚Üí make changes in your working directory.
2. Run `git diff` || `git diff --stat` || `git diff --color` (colourised output)  ‚Üí see unstaged changes.
3. Run `git add <file>` ‚Üí stage changes.
4. Run `git diff --cached` ‚Üí confirm staged changes.
5. Run `git commit -m "message"` ‚Üí commit confidently.

---

## ‚ö†Ô∏è Key Considerations
- **Output can be verbose**: Use flags like `--stat` for a summary (file names + line counts).
- **Binary files**: `git diff` won‚Äôt show meaningful diffs for binaries.
- **Colorized output**: Use `git diff --color` for readability.
- **Risk of missing changes**: Always check both `git diff` and `git diff --cached` before committing to avoid surprises.
